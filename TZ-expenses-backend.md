# ТЗ: бэкенд модуля «Расходы» (expenses)

Документ описывает функциональные требования к серверной части приложения для раздела заявок на расходы. Контракт согласован с текущим фронтендом (`/api/v1/expenses`, нормализация в `coerceExpense.ts`).

**Версия:** 1.0  
**Аудитория:** команда разработки tickets-back (или аналог).

### Подключение фронтенда к API (gateway, CORS, env)

Инструкция для разработчиков: **[docs/expenses-api-connection.md](./docs/expenses-api-connection.md)**.  
Каноническая копия в репозитории бэкенда: **`tickets-back/docs/expenses-frontend.md`** — при правках подключения обновляйте оба файла.

---

## 1. Цель

Реализовать (или довести до соответствия) API учёта заявок на расходы: создание черновиков, отправка на согласование, модерация (одобрение / отклонение / возврат на доработку), вложения, списки с фильтрами, расчёт и отдача сумм в UZS и эквивалента в USD.

Фронтенд не выполняет бизнес-валидацию статусов «на сервере» — источником истины остаётся бэкенд.

---

## 2. Роли и права доступа

| Действие | Кто |
|----------|-----|
| Просмотр списков и карточек заявок (в рамках своих прав на данные) | Роли просмотра раздела расходов (на фронте: любой пользователь с непустой ролью; на бэке — зафиксировать множество `ROLES_VIEW` или эквивалент). |
| Создание, редактирование черновика / на доработке, отправка, отзыв (если предусмотрено) | Автор / владелец заявки + правила по статусу. |
| Очередь «на согласовании», `approve` / `reject` / `revise` | **Только** роли модерации. На фронте совпадает с множеством: **Главный администратор**, **Администратор**, **Партнер** / **Партнёр** (регистронезависимое сравнение названия роли). Синхронизировать с `ROLES_MODERATE` в бэкенде. |
| Выгрузка отчёта | Те же роли, что и модерация (на фронте отчёт доступен вместе с очередью согласования). |

При отсутствии прав — **HTTP 403** с понятным `detail` / `message` в JSON.

---

## 3. Модель заявки (логическая)

### 3.1. Идентификатор

- `id` — строка или число, сериализуемое в JSON; фронт использует как строку в URL.

### 3.2. Обязательные и ключевые поля

| Поле | Описание |
|------|-----------|
| `description` | Текст описания расхода. |
| `expenseDate` | Дата расхода, формат **YYYY-MM-DD** (ISO date). |
| `paymentDeadline` | Опционально, крайний срок оплаты, **YYYY-MM-DD** или `null`. |
| `amountUzs` | Сумма в **сумах (UZS)** — число; в JSON допустима строка (Decimal), см. п. 8. |
| `exchangeRate` | Курс **UZS за 1 USD** на дату/для расчёта; число. |
| `equivalentAmount` | Эквивалент в **USD**; число. Должен быть согласован с правилами расчёта (п. 7). |
| `expenseType` | Код типа из фиксированного набора (см. п. 4). |
| `expenseSubtype` | Опционально, `string \| null`. |
| `isReimbursable` | Признак возмещаемости, `boolean`. |
| `paymentMethod` | Один из: `cash`, `card`, `transfer`, `other_payment` или `null`, если не указан. |
| `departmentId` | Опционально, `string \| null`. |
| `projectId` | Опционально, текст (на фронте — строка названия/идентификатора проекта). |
| `vendor` | Контрагент, `string \| null`. |
| `businessPurpose` | Цель / обоснование, `string \| null`. |
| `comment` | Комментарий, `string \| null`. |
| `status` | См. п. 5. |
| `createdByUserId`, `updatedByUserId` | Числа. |
| `createdBy` | Объект автора в ответах списка и `GET` по id: `id` (как `createdByUserId`), `displayName`, `email`, опционально `picture`, `position`. Данные из auth; если профиль недоступен — только `id`, остальное `null`. Допустимо `created_by` и `display_name` в snake_case. |
| `createdAt`, `updatedAt` | ISO datetime. |
| `submittedAt`, `approvedAt`, `rejectedAt`, `paidAt`, `closedAt`, `withdrawnAt` | ISO datetime или `null` по смыслу статуса. |
| `attachmentsCount` | Число вложений. |
| `attachments` | Опционально в ответе `GET` по id — массив вложений (п. 6). |

### 3.3. Имена полей в JSON

Допустимы **camelCase** (предпочтительно для нового API) или **snake_case**; фронт нормализует денежные поля: `amountUzs` / `amount_uzs`, `exchangeRate` / `exchange_rate`, `equivalentAmount` / `equivalent_amount`. Остальные поля желательно отдавать в одном стиле консистентно.

---

## 4. Типы расходов (`expenseType`)

Допустимые значения (как на фронте):

`transport`, `food`, `accommodation`, `purchase`, `services`, `entertainment`, `client_expense`, `other`

Невалидное значение при создании/обновлении — **422** с описанием ошибки.

---

## 5. Статусы и жизненный цикл

### 5.1. Значения `status`

`draft`, `pending_approval`, `revision_required`, `approved`, `rejected`, `paid`, `closed`, `not_reimbursable`, `withdrawn`

### 5.2. Ожидаемое поведение (минимум)

- **draft** — черновик, редактируется автором.
- **pending_approval** — на согласовании; доступна модерация.
- **revision_required** — возврат автору; редактирование по правилам продукта.
- **approved**, **paid**, **closed**, **rejected**, **not_reimbursable**, **withdrawn** — финальные или специальные состояния; переходы задокументировать в коде бэкенда.

### 5.3. Операции модерации

| Метод | Назначение |
|-------|------------|
| `POST .../submit` | Отправка на согласование (из draft / revision_required — по бизнес-правилам). |
| `POST .../approve` | Одобрение (роль модератора). |
| `POST .../reject` | Отклонение; тело JSON: `{ "reason": string }` (обязательная непустая причина). |
| `POST .../revise` | Возврат на доработку; тело: `{ "comment": string }` (обязательный комментарий). |
| `POST .../withdraw` | Отзыв заявки (если поддерживается продуктом). |

После операции возвращать **полный актуальный объект** заявки (как при `GET`).

---

## 6. Вложения

- **POST** `/api/v1/expenses/{id}/attachments` — `multipart/form-data`: поле `file`, опционально `attachmentKind` ∈ `payment_document`, `payment_receipt`.
- **DELETE** `/api/v1/expenses/{id}/attachments/{attId}`
- Ограничение размера на бэке: согласовать с фронтом (на фронте ориентир **15 МБ** на файл).
- Ответ после загрузки/удаления — обновлённая заявка (или явно задокументировать иной контракт).

Элемент вложения (логика):

- `id`, `expenseRequestId`, `fileName`, `storageKey`, `mimeType`, `sizeBytes`, `attachmentKind`, `uploadedByUserId`, `uploadedAt`

---

## 7. Деньги, валюта, эквивалент

- В БД и в API **основная сумма заявки** хранится в **UZS** (`amountUzs`).
- Поле `exchangeRate` — **сколько UZS за 1 USD** (как в форме и отчёте).
- Поле `equivalentAmount` — **эквивалент в USD**; бэкенд должен **вычислять и сохранять** (или гарантировать согласованность при сохранении), чтобы клиент не вызывал `.toFixed` на не-числах. Тип в JSON — число (предпочтительно), не строка без необходимости.
- Если клиент присылает сумму в иностранной валюте, сервер переводит в UZS по правилам продукта (курс ЦБ и т.д.) — фронт уже считает UZS для тела `create`/`update`; бэкенд валидирует и при необходимости пересчитывает.

---

## 8. Формат чисел в JSON

Фронт устойчив к:

- `number`
- строке с числом (в т.ч. с запятой)
- вложенным структурам вроде `{ "$numberDecimal": "..." }` или `{ "value": "..." }`

Рекомендация для нового кода бэкенда: отдавать денежные поля **как JSON number** в разумных пределах точности.

## 9. REST API (контракт)

Базовый префикс: **`/api/v1/expenses`**.

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/v1/expenses` | Список с пагинацией и фильтрами (п. 10). |
| POST | `/api/v1/expenses` | Создание заявки (тело как `ExpenseCreateBody` на фронте). |
| GET | `/api/v1/expenses/{id}` | Карточка заявки. |
| PUT | `/api/v1/expenses/{id}` | Обновление (частичное или полное — зафиксировать; фронт шлёт набор полей формы). |
| POST | `/api/v1/expenses/{id}/submit` | Отправка на согласование. |
| POST | `/api/v1/expenses/{id}/approve` | Одобрение. |
| POST | `/api/v1/expenses/{id}/reject` | Отклонение + `reason`. |
| POST | `/api/v1/expenses/{id}/revise` | На доработку + `comment`. |
| POST | `/api/v1/expenses/{id}/withdraw` | Отзыв. |
| POST | `/api/v1/expenses/{id}/attachments` | Загрузка файла. |
| DELETE | `/api/v1/expenses/{id}/attachments/{attId}` | Удаление вложения. |

Дополнительно используется фронтом:

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/v1/expense-types` | Справочник типов: `[{ code, label, sortOrder }]`. |
| GET | `/api/v1/projects` | Справочник проектов: `[{ id, name }]`. |
| GET | `/api/v1/exchange-rates?date=YYYY-MM-DD` | Опционально: курс для подстановки в форму (фронт также может брать курс с cbu.uz через прокси). Ответ: как минимум `date`, `rate`, `pairLabel`. |

Ответ списка:

```json
{
  "items": [ /* ExpenseRequest */ ],
  "total": 0,
  "skip": 0,
  "limit": 0
}
```

Ошибки: **4xx/5xx** с телом, откуда фронт читает `detail` или `message` (строка для показа пользователю).

## 10. Параметры списка GET `/api/v1/expenses`

Поддержать query-параметры (как на фронте):

| Параметр | Назначение |
|----------|------------|
| `status` | Фильтр по одному статусу (например `pending_approval` для очереди согласования). |
| `expenseType` | Код типа. |
| `isReimbursable` | `true` / `false`. |
| `dateFrom`, `dateTo` | Фильтр по `expenseDate`, формат **YYYY-MM-DD**. |
| `q` | Поиск по описанию / номеру (поведение уточнить). |
| `sortBy`, `sortOrder` | Сортировка (поля согласовать с UI). |
| `skip`, `limit` | Пагинация. |

**Замечание по продукту:** на фронте реестр «утверждённых» расходов сейчас дополнительно отфильтрован клиентом по статусам `approved`, `paid`, `closed`. Имеет смысл на бэкенде предусмотреть либо отдельный scope/query (например `scope=registry`), либо оставить единый список с фильтром `status` / несколькими статусами — если потребуется, расширить контракт (массив статусов).

---

## 11. Тело создания/обновления (согласование с фронтом)

Минимальный набор полей создания (`ExpenseCreateBody`):

- `description`, `expenseDate`, `amountUzs`, `exchangeRate`, `expenseType`, `isReimbursable`
- опционально: `paymentDeadline`, `expenseSubtype`, `paymentMethod`, `projectId`, `vendor`, `businessPurpose`, `comment`

Сервер обязан:

- валидировать обязательные поля и типы;
- выставлять `equivalentAmount` (и при необходимости пересчитывать `amountUzs`);
- корректно выставлять `status` при создании (черновик vs сразу на согласовании — по продукту).

---

## 12. Нефункциональные требования

- **Идемпотентность** там, где уместно (повторный `approve` — 409 или текущее состояние — зафиксировать).
- **Аудит**: кто и когда менял статус (поля `updatedByUserId`, метки времени).
- **Безопасность**: проверка владельца/роли на каждом endpoint; не отдавать чужие заявки без прав.
- **Производительность**: индексы по `status`, `expenseDate`, `createdByUserId` для списков и очереди согласования.
